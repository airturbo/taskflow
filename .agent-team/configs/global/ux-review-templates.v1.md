# 用户体验官模板

用于 `user_experience_officer` 在 `ux_review` 阶段输出标准化结论。

## 何时使用

在以下场景优先套用本文件：

- 最终体验评审 / final UX review
- 发布前完整走查 / release readiness UX check
- 高保真 Demo 对标评审 / parity review
- 团队需要明确的体验问题打回单与复审结论

默认输出顺序：

1. `UX_COVERAGE_MATRIX`
2. `UX_REVIEW_REPORT`
3. `UX_ISSUE_LOG`
4. `EXPERIENCE_ACCEPTANCE_NOTE`（仅在允许进入发布准备时产出；若阻断发布，可写明“暂不批准”）

## 严重级别定义

| Level | 含义 | 处理要求 |
|---|---|---|
| P0 | 致命体验问题：核心路径无法完成、强烈误导、明显破坏信任或严重掉档次 | 必须阻断发布 |
| P1 | 高优先级问题：核心路径可完成但明显别扭、低效、混乱、不成熟 | 默认阻断发布，除非用户明确接受风险 |
| P2 | 中优先级问题：不阻断主路径，但明显影响完成度、一致性、理解成本 | 进入迭代清单，建议在下一轮修复 |
| P3 | 低优先级问题：细节 polish、措辞、间距、动效、边角行为 | 可累计处理，但不能长期忽略 |

## 模板一：UX_COVERAGE_MATRIX

```md
# UX_COVERAGE_MATRIX — {项目/版本/构建号}

## 1. 覆盖摘要
- release scope 功能总数：{N}
- 已完整体验：{N}
- 存在问题：{N}
- 未纳入本轮：{N}
- 覆盖结论：完整 | 基本完整但有缺口 | 不完整

## 2. 用户剧本
- 剧本 A：{首次用户目标}
- 剧本 B：{回访用户目标}
- 剧本 C：{重度用户目标}
- 其他补充剧本：{异常恢复 / 协作 / 设置 / 导入导出 / 跨视图}

## 3. 覆盖矩阵
| Feature ID | Area | Journey | Persona | Trigger | Interaction Check | Result Check | Evidence | Verdict | Issue IDs | Next Action | Owner |
|---|---|---|---|---|---|---|---|---|---|---|---|
| F-001 | {模块} | {用户旅程} | {用户类型} | {按钮/快捷键/拖拽/菜单} | {反馈是否明确} | {结果是否正确且可理解} | {录屏/截图/自动化日志路径} | pass / issue / blocked / not_in_scope / waived_with_risk_ack | {UX-001} | {修复/补测/豁免说明} | {角色} |

## 4. 覆盖缺口
- {哪些功能还未体验，为什么}

## 5. 复审重点
- {下轮必须回放的功能/旅程}
```

`Verdict` 建议只用以下值：

- `pass`
- `issue`
- `blocked`
- `not_in_scope`
- `waived_with_risk_ack`

## 模板三：UX_ISSUE_LOG

```md
# UX_ISSUE_LOG — {项目/版本/构建号}

| Issue ID | Level | Status | Area | Journey | Symptom | Repro Steps | Evidence | Expected | Actual | Impact | Suggested Fix | Next Action | Problem Owner | Delivery Owner | Recheck Owner |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|---|
| UX-001 | P1 | open | 列表页 | 创建任务 | 按钮主次混乱 | 1.进入列表页 2.点击创建 | {截图/录屏/自动化日志} | 主按钮应唯一且稳定 | 页面上有两个等权主操作 | 用户犹豫，降低转化 | 合并为单一主操作按钮并统一文案 | 产品先更新验收标准，再回归创建链路 | 产品经理 | 前端 | 用户体验官 |
```

状态建议只用以下值，避免混乱：

- `open`
- `in_progress`
- `ready_for_recheck`
- `closed`
- `wont_fix_with_risk_ack`

Issue ID 建议统一为 `UX-001`、`UX-002` 递增。

Owner 默认约定：
- `Problem Owner`：默认由产品经理承接问题定义与验收口径
- `Delivery Owner`：默认由项目经理协调后指定设计 / 前端 / 后端 / QA 等执行角色
- `Recheck Owner`：默认由 `user_experience_officer` 负责最终复验

## 模板四：EXPERIENCE_ACCEPTANCE_NOTE

```md
# EXPERIENCE_ACCEPTANCE_NOTE — {项目/版本/构建号}

## 1. 验收结论
- 是否通过：通过 | 不通过
- 结论日期：{YYYY-MM-DD}
- 评审角色：用户体验官
- 对应覆盖矩阵：{UX_COVERAGE_MATRIX 文件或 ID}
- 对应评审报告：{UX_REVIEW_REPORT 文件或 ID}

## 2. 放行依据
- 已关闭的关键问题：{列表}
- 剩余风险：{如无则写“无阻断级风险”}
- 可接受遗留项：
  - {P2/P3 问题 + 为什么可接受}
- 覆盖完整度说明：{为什么可以认为 scope 已完整体验}

## 3. 发布建议
- 是否允许进入 `release_preparation`：是 | 否
- 发布后重点观察：
  - {指标/用户反馈/客服信号/行为漏斗}
- 需要继续跟踪的功能：
  - {列表}

## 4. 备注
- {任何需要项目经理、产品经理继续跟踪的事项}
```

## 评审纪律

1. 先做真实前台体验，再对照文档，不要只看产物文字、代码或截图。
2. 每个高优先级问题都必须给出：发生位置、影响人群、复现路径、建议改法。
3. 不要用“感觉不好”这种空话，必须把问题说到可以执行。
4. 若结论是“阻断发布”，必须点名回退给 `product_manager` 和 `project_manager_orchestrator`。
5. 若结论是“允许进入发布准备”，也必须写清剩余风险和观察点。
6. 评价标准偏高是故意的：目标不是“做完”，而是“像一个成熟产品”。
7. 如果没有 `UX_COVERAGE_MATRIX`，就不应声称“所有功能都体验完整”。
8. 本地执行时优先引用 `.agent-team/configs/global/` 下的 `.v1` 文件，避免口径漂移。
