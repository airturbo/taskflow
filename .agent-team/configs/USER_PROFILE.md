## 基本信息

- **称呼**：master
- **角色**：产品经理，负责 TaskFlow 产品的整体方向与质量把控
- **核心目标**：把 TaskFlow 做成对标 Todoist / Things 3 水准、真正可上架的产品

---

## 沟通风格

### 他喜欢的方式
- **结论先行**：先说结论，再说原因，不要绕弯子
- **简短推进**：「继续」「下一步」「两者都做」这类简短指令就是让你接着做，直接执行
- **主动汇报**：做完了主动说结果，不等他追问
- **有判断力**：遇到两可的情况，先分析利弊，然后给出你自己的推荐意见，不把所有选择都甩给他
- **附图分析**：他发截图时，主动对比前后、识别根因，不等他逐条描述问题

### 他不接受的方式
- 废话和空洞的礼貌回复
- 「这两种方案各有优缺点，请您选择」式的推卸判断
- 修复了表面但没解决根因的补丁
- 反复修复失败（第三次失败就应该考虑架构层重构）
- 只做「及格」而不追求「完成度高、细节一致、真实可用」

## 审美与设计偏好

### 基准标准
对标 **Todoist / Things 3**：极简、精致、专业，不接受模板感和过度工程化的 UI。

### 具体要求（逐条遵守）

**布局与层级**
- CRAP 原则是底线：每个界面的对比、重复、对齐、亲密性必须经过检查
- 父子层级必须视觉可区分：文件夹用粗体/深色，子清单用细体/浅色/缩进
- 数字列必须垂直对齐（同一列的数字在同一个 x 坐标上）
- 分组之间的间距必须大于组内间距（至少 2 倍）

**交互克制**
- 入口原则：「如无必要勿增实体」，能用图标就不用文字，能折叠就折叠
- 正式创建入口要显式，不要隐式点击创建
- 功能操作（编辑/删除）hover 后再显示，不要始终占用视觉空间

**细节完整性**
- 滚动条：完全隐藏，内容可以滚动但不显示轨道
- 抽屉/弹层：必须有遮罩（不能出现"抽屉开着但旁边一片空白"的情况）
- 图标风格：同一区域内图标风格必须统一，不能 emoji 和 SVG 混用
- 计划/DDL 标签：在标题和数字之间展示，不要独占一行

**响应式**
- 手机（≤680px）：底部 Sheet，下滑关闭，92vh 最大高度
- 平板/中屏（680-1280px）：右侧 400px 抽屉，点任务自动打开
- 宽屏（>1280px）：固定右侧 rail，无需抽屉

## 决策风格

- **渐进主义**：先做最小可验证的版本，验证后再扩展（如「先用 Supabase 验证 PMF」）
- **不过度工程化**：能简单解决的不做复杂架构，但一旦遇到架构性问题会果断重构
- **截图驱动**：发截图意味着要对比分析和修复，不是单纯展示
- **信任判断**：愿意听专业建议，但最终决策权在自己，不接受「你来选」

---

## Bug 处理规范

1. **截图/视频发来后**：主动对比前后，说清楚根因
2. **第一次修复**：可以先打定向补丁
3. **第二次修复同一问题**：必须说清楚为什么上次没解决，这次从根本上怎么改
4. **第三次修复同一问题**：升级到架构层面重构，停止打补丁

## 历史上重要的对话结论（2026-04-09）

| 主题 | 结论 |
|------|------|
| 后端选型 | Supabase 而非 Firebase，原因：关系型更适合、国内可访问、PG 可迁移 |
| 部署方案 | Vercel（自动 CI/CD）+ Supabase（数据层），先跑 Web 版验证 |
| 响应式策略 | 手机 Sheet / 平板右侧抽屉 / 桌面固定 rail，按尺寸分级 |
| 侧边栏交互 | 折叠模式用 click-toggle，不用 hover（hover 容易误触） |
| 滚动条 | 完全隐藏，参考 Things 3 |
| 文件夹层级 | 文件夹（粗体+方形色点）> 子清单（细体+圆形色点+缩进），视觉必须可区分 |
| 设计原则沉淀 | CRAP/Gestalt 原则已注入 PM/UI/FE 三个角色的 system_prompt |
