## 1. 本轮目标

围绕“双时间语义”完成一次从产品定义到前台实现再到体验复查的完整闭环：

> 在不膨胀导航结构的前提下，让任务同时支持“计划完成时间”和“硬性 DDL”，并让列表、详情、日历、时间线、筛选、提醒与统计都说同一种时间语言。

---

## 2. 必须完成（R1）
- PRD v4.0.0 正式明确双时间语义与信息架构边界；
- Web / Desktop 持久化链路新增 `deadlineAt`；
- 详情面板支持开始时间 / 计划完成 / 硬性 DDL；
- 今日 / 未来 7 天筛选改为单行切换“计划 / DDL”；
- 列表 / 看板 / 四象限 / 日历 / 时间线完成最小可用的双时间表达；
- 构建通过并完成一轮体验官 focused 复审；
- 若体验官发现明显问题，项目团队需立即在同轮收口。

## 4. 实现边界

### 4.1 本轮直接落地
- `dueAt` 继续作为内部“计划完成时间”承载字段；
- 新增 `deadlineAt` 承载硬性 DDL；
- 现有 `startAt` 不变，继续服务时间线 / 日历区间；
- 系统视图先改造“今日 / 未来 7 天”两条时间型入口；
- 保存筛选的更复杂时间切换留作下一轮扩展。

### 4.2 本轮不做
- 自然语言双时间解析；
- 新增大量风险视图；
- 复杂批量编辑；
- 全量设计系统重绘。

---

## 5. 节奏要求
- 先让时间语义诚实，再继续做风险高级能力；
- 先确保同类筛选单行切换成立，再考虑更多时间型视图；
- 先让前台真实可用，再决定是否继续扩范围。

---

## 6. 验收否决条件
以下任一情况都不视为本轮完成：
- 只是把“截止时间”文案改名，但没有新增硬性 DDL；
- 导航膨胀成双份时间筛选；
- 列表卡片与详情面板仍让人分不清计划和 DDL；
- 时间线或日历因双时间改造而失去主要可读性；
- Web / Desktop 任一持久化链路丢字段或构建失败。
