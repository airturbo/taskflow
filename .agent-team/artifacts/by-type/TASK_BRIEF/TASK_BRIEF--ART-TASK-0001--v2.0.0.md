## 1. 项目目标

根据用户最新决策，当前项目不再修补既有 MVP，而是执行一次**方向回正型重构**：

> 以滴答清单为对照对象，重新设计 PRD 与交互，移除商业化限制，将高级功能默认开放，并由项目经理重新编排角色，把项目推进到一个**高保真、可交互、可验收的 Web Demo**。

这里的“可用 Demo”定义不是“能点开就算完成”，而是：
- 核心任务链路完整
- 核心视图真实可操作
- 习惯 / 专注 / 统计具备演示价值
- 对齐评审可清晰说明哪些已对齐、哪些仍未对齐

---

## 2. 本轮必须完成（R1）
- 重建 PRD 与功能对齐边界
- 重建设计规范与关键交互流
- 输出一份明确的功能对齐评审
- 重写 `web/` 演示端，使其从 MVP 升级为 parity demo
- 完成构建验证与本地预览交付

## 3. 本轮保留但允许部分降级（R2）
- 真实多人协作
- 真实云同步
- 真实浏览器推送与位置提醒
- 第三方日历/平台双向同步
- 原生端能力（锁屏浮窗、小组件、系统级提醒）

原则：
**R2 不是不做设计，而是不要求本轮接入真实外部依赖。**

## 5. 交付定义

### 5.1 Demo 完成标准
1. 默认打开即为新工作台，不再出现旧 MVP 口径文案。
2. 任务系统支持高级属性与详情编辑。
3. 至少四个滴答式核心视图有真实交互。
4. 习惯 / 专注 / 统计三条工具链可演示。
5. 信息架构、视觉骨架、交互层级明显向滴答清单对齐。
6. 有一份明确的对齐评审说明完成度。
7. 本地预览可打开，`npm run build` 成功。

### 5.2 不视为完成的情况
- 只有漂亮壳子，没有真实数据操作
- 仍然停留在“列表 + 看板 + 番茄”的旧 MVP 形态
- 只有文档，没有可运行 Demo
- 只有功能堆叠，没有对齐评审

---

## 6. 风险与决策

| 风险 | 决策 |
|------|------|
| 再次被“先做简版”拉回 MVP | 本轮不接受降级为简单待办应用 |
| 一步做到生产级过于庞大 | 交付线设为高保真可交互 Demo，不假装已完成后端能力 |
| 旧代码会拖累节奏 | 优先重写核心工作台，而不是继续补丁式扩展 |
| 设计/实现脱节 | 先出 PRD、UI、UX、对齐评审，再推进代码 |

---

## 7. 节奏要求
- 先定范围，再定交互，再落实现
- 每完成一层设计，都要能服务于 Demo 实现，不做纯文档堆砌
- 项目经理持续以“是否更接近 TickTick”作为唯一判断标准
