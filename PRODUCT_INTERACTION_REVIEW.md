# TaskFlow 产品交互诊断报告

**审查日期**: 2026-04-12  
**审查视角**: 产品体验（非工程实现）  
**产品定位**: 多平台任务管理（Web / Desktop / Mobile / Obsidian）  
**核心判断**: 产品有良好的技术骨架，但多处设计把系统内部概念直接暴露给用户，违反了"不让用户思考"的基本原则

---

## 🔴 一级缺陷：认知负担过载（Cognitive Overload）

### 1. 三时间字段是设计失误，不是功能优势

**现状**：`startAt` / `dueAt` / `deadlineAt` 三个字段并存，用户必须理解：
- "计划完成" vs "硬性截止" 的区别
- 何时填 dueAt，何时填 deadlineAt
- 为什么会有两种不同的"完成时间"

**根因**：设计者把系统内部实现泄漏给了用户。正常人的认知模型里只有一个"截止日期"。Things 3 成功的核心原因之一就是时间字段极度克制。

**建议**：
- 默认只暴露一个"截止时间"字段
- `startAt` 改名为"开始时间"并折叠到高级选项
- `dueAt` / `deadlineAt` 合并为一个用户可见字段，通过 workspace 设置决定底层用哪个驱动排序——这是内部决策，不该暴露给用户

---

### 2. 四象限矩阵的实现逻辑错误

**现状**：Q1 = `tag-urgent` + `tag-important`，通过**标签**决定象限位置。

**致命问题**：
- 用户拖拽任务到不同象限 → 实际上是在**修改标签**，而不是改变任务的本质属性
- 如果用户在列表视图删掉了 `tag-urgent` 标签，任务会在矩阵中自动跑到别的象限——用户毫不知情
- 标签应用于分类（项目、情境），优先级/紧急度应该是独立属性
- 这是数据模型层面的设计错误，不是 UI 问题

**建议**：增加 `urgency` 和 `importance` 两个独立字段（boolean 或 0-1 分值），矩阵视图读这两个字段，与标签系统完全解耦。

---

### 3. 视图之间缺乏状态一致性感知

**现状**：产品有 6 种视图（List/Calendar/Kanban/Timeline/Matrix/Stats），切换时：
- 用户无法感知"我在哪个任务上"（无全局选中态持久化）
- 过滤条件/搜索不跨视图保留（切换到 Calendar 后搜索消失）
- Kanban 的状态列（Todo/Doing/Done）与 List 的 status 字段是映射关系，但两处的修改路径完全不同，造成认知断裂

**建议**：实现"任务上下文持久化"——当前选中任务、当前搜索词、当前过滤器应在视图切换时保留。用户应该感觉在"同一工作空间的不同视角"，而非"进入了另一个不同的应用"。

---

## 🟠 二级缺陷：交互模型不一致

### 4. 完成流程有二次确认摩擦

**现状**：移动端点击完成 → 弹出 `MobileConfirmSheet` 确认对话框 → 再点一次 → 任务完成。

**问题**：每次完成任务都要两次点击，直接摧毁了打勾的满足感。GTD/任务管理的核心爽点就是「一键完成」的即时反馈。Todoist、Things 3 都是直接勾选，配合可撤销的 Toast 作为安全网，不会在完成路径上设置障碍。

**建议**：
- 移除完成确认对话框
- 改为 Undo Toast（代码已有实现，3 秒可撤）
- 确认对话框只在**永久删除**这类不可逆操作时使用

---

### 5. NLP 智能输入没有视觉反馈

**现状**：`smart-entry.ts` 实现了 NLP 解析，支持"明天下午3点打电话 #工作 !高"这样的自然语言输入。

**问题**：
- 用户输入过程中看不到解析结果预览
- 不知道哪些词被识别了，哪些没有
- 识别失败时没有任何提示，任务被当成纯文本存储
- 用户如何知道这个功能存在？

对比 Linear 的 issue 创建、Notion 的日期识别——都有实时高亮显示识别结果。

**建议**：
- 输入框下方实时展示解析预览卡片
- 高亮显示已识别的日期/标签/优先级："明天下午3点"变成蓝色 chip，"#工作"变成对应颜色的标签 chip
- 让用户看到 AI 在工作，建立信任感，同时让功能本身可被发现

---

### 6. 移动端 Focus Tab 分组逻辑对用户不透明

**现状**：分组为 🔴逾期 / 📌今天计划 / ⚡今天到期 / 📥收件箱 / 📅即将

**问题**：
- "今天计划"（dueAt）vs "今天到期"（deadlineAt）—— 两个用户看起来相同的概念分成了两组，有割裂感
- "收件箱"混在时间分组里，是不同维度的概念（无归属 vs 时间属性），维度混搭
- 用户不理解为什么一个任务在"计划"组而不在"到期"组

**建议**：简化分组模型，参考 Fantastical 的"今天/明天/以后"三段模型，或 Things 3 的"到期今天/开始今天/收件箱"，保持维度统一，不要把时间维度和分类维度混合。

---

## 🟡 三级缺陷：功能完整性缺口

### 7. 重复任务的用户感知断裂

**现状**：完成一个重复任务后 → 系统自动创建下一个实例，旧任务标记为完成。

**问题**：
- 用户不知道刚才完成的任务会"复活"
- 下一个实例出现时，用户可能困惑"这个任务我不是刚完成了吗"
- 没有视觉区分告知"这是一个重复任务的第 N 次执行"

**建议**：
- 重复任务在卡片上应有持续可见的标识（循环图标）
- 完成时的 Toast 应说明"已安排下次重复：4月19日"
- 建立用户对重复任务生命周期的正确心理模型

---

### 8. Kanban 看板缺少 WIP Limit（在制品限制）

**现状**：看板（Todo/Doing/Done）已实现，但没有在制品限制的概念。

**问题**：没有 WIP Limit 的看板只是一个"三列任务列表"，失去了看板管理方法论的核心价值。用户把 20 个任务拖进"进行中"，看板变成另一个待办列表，没有任何管控意义。

**建议**：允许用户设置每列的 WIP Limit，超出时视觉警示（列头变红/数字变红）。这是产品真正体现专业性的地方。

---

### 9. Stats 视图的洞察没有可操作出口

**现状**：Stats 视图提供了"未排期恢复计划"和"截止日期冲突分析"的分析能力。

**问题**：分析完了然后呢？用户看到"下周有3个任务冲突"后，需要手动回到列表视图逐个修改。洞察和行动之间没有桥梁，这个视图只是"让用户焦虑"，而非"帮用户解决问题"。

**建议**：Stats 视图的每个洞察旁边应该有"修复"或"重新排期"按钮，直接触发批量操作，把分析转化为行动。

---

## ⚪ 产品战略级问题

### 10. 产品定位模糊：个人效率工具 vs 团队协作工具

**现状**：
- 代码中有 `assignee`、`collaborators[]`、`comments[]` —— 明显有协作设计意图
- 同时又有：本地优先存储、Obsidian 插件、单人 workspace —— 典型的个人工具特征

**当前状态是两头不靠**：
- 协作功能太弱（没有权限系统、没有通知、没有 @mention 跳转）
- 个人功能被协作字段稀释（25 个字段里很多是团队场景）

**建议**：先把个人效率做到极致（对标 Things 3 的专注度），再考虑轻量协作。一个人用得爽的工具，才可能被推荐给团队用。

---

### 11. 跨平台功能集差异缺乏明确策略

**现状**：
- Mobile 缺少 Calendar/Timeline/Kanban 视图
- Desktop 有文件附件，Mobile 没有
- Obsidian 插件功能集不明确
- 功能集的差异没有在 UI 层做明确的适配引导

**边界情况未处理**：Desktop 创建了带文件附件的任务，Mobile 上如何显示？这类跨平台边界情况没有清晰处理策略，容易造成数据显示异常。

## 总结

这是一个**有工程师视角、缺产品视角**的产品。技术实现扎实（多平台共享 core、本地优先存储、虚拟滚动等），但多处设计把"系统内部概念"直接暴露给用户。

**修复 P0 的三个问题**（时间字段合并、去掉完成确认、NLP 实时反馈），体验可以有质的提升，边际成本极低。

**P1 的四象限逻辑错误**是潜伏的数据层 bug，需要在用户规模扩大前修复，否则会产生大量脏数据。
