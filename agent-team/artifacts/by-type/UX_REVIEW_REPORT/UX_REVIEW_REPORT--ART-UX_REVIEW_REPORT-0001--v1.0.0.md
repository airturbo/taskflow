- **深色视觉基调统一。** 颜色、层级、圆角和边框语言基本一致，已经不是粗糙模板页。

## 5. 关键断点
- **断点 1：快速创建的自然语言解析会直接污染任务标题。** 这是主入口级别的问题，用户第一次试就会对产品可信度打折。
- **断点 2：创建后强制跳转所属清单，破坏上下文连续性。** 用户明明在“今日”工作，系统却把人拖去“收件箱”，这是典型的工作流打断。
- **断点 3：完成态过滤口径不一致。** 侧栏计数说“没有了”，主区却还在显示，属于会破坏信任的设计错误。
- **断点 4：多视图在低密度场景下完成度不足。** 日历、时间线、四象限都有结构，但当任务少时，主画面信息量塌得太快，容易显得空和薄。
- **断点 5：专注主任务与当前详情任务不同步。** 一旦用户开始计时却不知道计在哪条任务上，专注链路就不再可靠。

## 8. 修改建议（按优先级）

### 第一优先级：先把“可信度问题”修掉
1. **修快速创建解析。** 这是最前面的一刀，不能让用户第一次录入就看到脏标题。建议直接从 `web/src/utils/smart-entry.ts` 下手，先做时间片段 token 化，再做标题回拼，不要用容易漏字符的粗暴替换。
2. **修创建后的上下文保持。** `web/src/App.tsx` 的 `createTask()` 里现在会在非 list 视图下强制 `setActiveSelection(list:${listId})`。这个策略应该撤掉。正确做法是：任务归属可以是收件箱，但用户仍留在“今日”，系统给一个“已创建并选中”的明确反馈。
3. **统一 completed 口径。** 产品先定规则：行动视图默认只看未完成，还是展示完成但折叠。规则一旦定下来，`countsBySelection` 与 `visibleTasks` 必须统一，不允许左侧和中间各说各话。

### 第二优先级：把多视图从“有结构”补到“能工作”
4. **为日历 / 时间线 / 四象限补 sparse-state 设计。** 任务少的时候，不能只剩一块大黑底和一个小卡片。至少要补：当前时间范围摘要、任务总量、下一步建议、空态解释、负荷感表达。
5. **让番茄主任务来源更清楚。** 如果 focus 工具可以独立于当前详情任务存在，就必须显式告诉用户“你现在在为哪条任务计时”；如果不是，就把主区和右栏强绑定。

### 第三优先级：补发布前体验门禁的剩余证据
6. **补测已完成 / 回收站 / 标签 / 智能清单。** 这不是可有可无的“锦上添花”，而是体验官放行前的必要证据。否则当前评审只能给“阻断发布”。
