---

## 1. 背景

当前产品把“我准备什么时候做完”和“这件事最晚绝不能拖到什么时候”混在一个截止时间里。结果是：

1. 用户为了保留真实 DDL，不得不牺牲自己的内部计划；
2. 用户为了给自己排期，又会把真实硬截止覆盖掉；
3. 今日 / 未来 7 天 / 逾期 / 日历 / 时间线都只看一个时间，导致筛选语义与风险提示失真；
4. 任务卡片、详情面板、提醒和统计无法同时表达“执行落点”与“风险边界”。

这不是字段命名问题，而是产品时间模型不够诚实。

---

## 2. 产品目标

### 2.1 本轮目标
- 让任务同时承载 **执行计划** 与 **外部约束** 两种时间语义；
- 保持导航结构克制，不因双时间而复制一整套筛选入口；
- 让列表、详情、日历、时间线、提醒与统计都能区分“计划完成”和“硬性 DDL”；
- 在不破坏现有排期能力的前提下，为后续风险管理、提醒升级和时间复盘留出演进空间。

### 2.2 一句话定义

> **计划完成时间是执行主时间，硬性 DDL 是风险约束时间。**

---

## 3. 关键概念与语义

### 3.1 计划完成时间（Planned Finish）
- 含义：用户计划在何时完成该任务；
- 属性：可调整、可拖动、用于日常排期；
- 作用：决定列表的主时间展示、日历落点、时间线条形块、默认时间筛选口径；
- 旧模型映射：当前系统里的 `dueAt` 迁移为“计划完成时间”语义。

### 3.2 硬性 DDL（Hard Deadline）
- 含义：不可违背的外部截止边界；
- 属性：更稳定、风险语义更强、优先用于告警；
- 作用：决定 DDL 视角筛选、风险提示、冲突校验、最终到期提醒；
- 新增字段：独立于计划完成时间存在。

### 3.3 开始时间（Start Time）
- 含义：任务执行区间的开始时刻；
- 作用：继续服务时间线与日历中的时间块排布；
- 与双时间关系：不替代计划完成或硬性 DDL，只负责描述“从什么时候开始做”。

---

## 4. 范围定义

### 4.1 本轮必须覆盖（R1）
1. 任务数据模型新增“硬性 DDL”；
2. 详情面板支持编辑：开始时间 / 计划完成 / 硬性 DDL；
3. 列表、看板、四象限卡片支持同时表达主时间与 DDL 风险提示；
4. 日历继续以计划完成为主落点，并能识别存在 DDL 的任务；
5. 时间线继续以开始时间 + 计划完成构成时间条，并能显示 DDL 标记；
6. 今日 / 未来 7 天筛选改为单行入口内切换“计划 / DDL”；
7. 统计、逾期判断、提醒逻辑同步适配双时间语义；
8. Web 与 Desktop Repository 持久化链路都能保存新字段。

### 4.2 本轮允许暂不覆盖（R2）
- 自然语言一次输入同时解析“计划完成 + 硬性 DDL”；
- 智能清单编辑器里的双时间切换 UI；
- 更复杂的风险等级面板（如撞线趋势、缓冲期建议）；
- Deadline 专属批量编辑与独立提醒模板库。

---

## 5. 信息架构原则

### 5.1 不复制导航
双时间语义成立，但导航不新增两套平行入口。

**错误方案：**
- 今日计划完成
- 今日硬截止
- 未来 7 天计划完成
- 未来 7 天硬截止

这会让系统视图横向膨胀、学习成本翻倍。

**正确方案：**
- 今日 ｜ `计划 / DDL`
- 未来 7 天 ｜ `计划 / DDL`

规则：
- 默认落在 **计划**；
- 用户需要时，再在当前行内切到 **DDL**；
- 当前行的计数、排序、空状态和结果列表都跟随当前模式切换；
- 切换是“同一筛选的不同时间口径”，不是“新建一条平行筛选”。

### 5.2 同类新增筛选同理
后续若新增“逾期 / 风险 / 本周”等时间型筛选，也必须遵守同一规则：
- 同一行或同一入口内切换；
- 默认计划完成；
- 按需切到 DDL；
- 不允许让侧边导航因为时间语义而倍增。

---

## 6. 交互设计

### 6.1 详情面板
时间区块从旧的“开始时间 + 截止时间”改为：
- 开始时间
- 计划完成
- 硬性 DDL

交互规则：
1. 计划完成是主字段，视觉更靠前；
2. 硬性 DDL 作为风险字段，文案必须明确写“硬性 DDL”；
3. 当 `计划完成 > 硬性DDL` 时，立即显示冲突提示；
4. 若任务只有 DDL 没有计划完成，允许保存，但提示“建议补一个计划完成时间”；
5. 若只有计划完成没有 DDL，不视为错误。

### 6.2 列表 / 看板 / 四象限卡片
- 主时间位展示：**计划完成**；
- 若存在硬性 DDL，则以小 badge 或次级文案提示；
- 当计划完成晚于 DDL 时，卡片提升风险优先级并显示冲突样式；
- 当只有 DDL 没有计划完成时，卡片允许直接显示 DDL 作为当前唯一时间信息；
- 不允许在普通任务卡片上铺两行大时间，避免视觉噪音。

### 6.3 日历
- 日历的落点继续由 **计划完成 / 开始时间** 驱动；
- 硬性 DDL 不生成第二个主落点，不额外复制一张“DDL 日历”；
- 存在 DDL 的任务在日历卡片上显示轻量风险标记（点、tag 或微 badge）；
- 日历拖拽改期默认调整计划完成与开始时间，不直接改动硬性 DDL。

### 6.4 时间线
- 时间条继续表达“开始时间 → 计划完成”；
- 硬性 DDL 以细标记 / marker 方式叠加到时间线上；
- 当 DDL 落在线程窗口内但早于计划完成时，要能看出“条已撞线”；
- 时间线拖拽、拉伸默认改计划区间，不直接篡改硬性 DDL。

### 6.5 系统筛选
- 今日、未来 7 天采用同一行内切换；
- 默认计划完成；
- 用户切到 DDL 后，该行的 count、标题状态、结果集同步切换；
- DDL 状态在视觉上更强，但仍保持克制，不做整行高饱和红底。

---

## 7. 提醒与统计

### 7.1 提醒
- 自动排期提醒继续锚定开始时间 / 计划完成；
- 任务到期提醒默认优先锚定硬性 DDL；若无 DDL，再退回计划完成；
- 提醒文案必须能分清“计划完成提醒”与“DDL 已到”。

### 7.2 统计
本轮至少保留以下统计能力：
- 当前结果总数
- 活跃 / 已完成
- 逾期（优先按硬性 DDL 判定；无 DDL 时回退计划完成）
- 已排期

后续可继续扩展：
- 仅计划逾期
- DDL 临近
- 计划撞线
- 无计划但有 DDL

---

## 8. 数据模型

### 8.1 任务字段
- `startAt`: 开始时间
- `dueAt`: 兼容现有存量实现，当前版本内部继续承载“计划完成时间”
- `deadlineAt`: 新增字段，承载硬性 DDL

### 8.2 视图状态
新增时间口径状态，用于今日 / 未来 7 天等时间型筛选：
- `planned`
- `deadline`

该状态应持久化到 Web localStorage 与 Desktop workspace shell，保证刷新后用户仍停留在上次选中的口径。

---

## 9. 验收标准

### 9.1 功能验收
1. 任务详情能同时编辑开始时间、计划完成、硬性 DDL；
2. 列表 / 看板 / 四象限卡片能表达双时间语义且不显乱；
3. 今日 / 未来 7 天侧边筛选支持单行内切换“计划 / DDL”；
4. 切到 DDL 后，计数与结果集确实变化；
5. 日历仍按计划时间落点，时间线仍按计划区间渲染；
6. 时间线中存在 DDL 的任务可看到风险标记；
7. 任务持久化、刷新、桌面仓储查询不丢字段；
8. `npm run build` 成功。

### 9.2 体验验收
1. 用户能一眼分清“计划完成”和“硬性 DDL”；
2. 导航没有膨胀成双份时间筛选；
3. 卡片主次层级清晰，不出现双时间堆砌；
4. 详情面板冲突提示明确，但不过度制造焦虑；
5. 日历 / 时间线保持克制，不因第二时间语义而炸界面。
