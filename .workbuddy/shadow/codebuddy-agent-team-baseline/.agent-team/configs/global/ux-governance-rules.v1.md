# 用户体验治理规则

用于 `user_experience_officer` 在 `ux_review` 阶段执行覆盖完整、可追责、可复审的最终体验治理。

## 治理目标

把“体验评审”从主观挑刺，升级为一套带覆盖边界、评判口径、问题流转和发布门禁的正式治理机制。

默认目标不是证明产品“差不多能用”，而是确认它是否已经做到：

- 真实用户能顺畅完成目标任务
- 所有发布范围内功能都被完整体验过
- 每个功能的“触发 → 交互 → 结果”都被核查过
- 问题有评价、有 owner、有后续处理路径
- 未达到门槛时，能够明确阻断发布并组织返工

## 核心原则

1. **真实用户优先，不做展台式浏览**
   - 必须按真实用户任务去操作，而不是只看页面摆得是否好看。
   - 评审时默认至少覆盖：首次使用者、回访用户、高频重度用户三类视角。

2. **功能全覆盖，不允许只抽查主页面**
   - 凡是本轮 release scope 内、用户可见或可触发的功能，都必须进入覆盖账本。
   - 包括主功能、次级入口、设置项、空态入口、失败重试、批量操作、快捷键、上下文菜单、拖拽、导入导出等。

3. **逐项核对 Trigger → Interaction → Result**
   - Trigger：用户如何触发它。
   - Interaction：系统如何接住用户操作。
   - Result：结果是否正确、清楚、可信、可恢复。
   - 三者任一环节明显掉档次，都属于体验问题。

4. **评价必须可执行**
   - 每个问题都要说清：发生位置、受影响人群、复现方式、预期结果、实际结果、建议处理、责任角色。
   - 不允许只写“感觉不高级”“体验一般”。

5. **治理结果必须闭环**
   - 体验问题不是意见箱，而是要进入返工、复审、放行决策的治理输入。
   - P0 / P1 问题默认阻断发布，直到关闭或被明确风险签收。

## 适用范围

以下内容默认纳入本规则：

- 本轮发布或演示范围内的全部用户可见功能
- 所有核心用户旅程
- 关键状态：默认态、hover、focus、active、loading、success、error、empty、disabled、undo、conflict
- 所有已声明支持的交互方式
- 所有会影响用户判断、效率、信任感或完成度的结果反馈

若某功能明确不在本轮范围，必须在 `UX_COVERAGE_MATRIX` 中显式标注 `not_in_scope`，不能默默跳过。

## 基本定义

| 术语 | 定义 |
|---|---|
| 用户旅程 | 用户为了完成一个目标而经历的完整任务链路 |
| Trigger | 用户发起动作的入口，如按钮、快捷键、拖拽、上下文菜单、通知跳转 |
| Interaction | 系统在过程中给出的反馈、状态变化、引导与容错 |
| Result | 用户最终看到的结果，包括是否成功、是否清楚、是否可信、是否可恢复 |
| Coverage | 是否已对某功能/链路完成真实操作与评价记录 |
| Risk Ack | 对未修复问题的明确风险接受，必须有责任角色和后续计划 |

## 强制输入

开始治理前，至少要具备：

1. 可访问的最新预览地址 / Demo / 构建产物
2. 最新 `PRD`
3. 最新 `UI_SPEC`
4. 最新 `UX_FLOW`
5. 最新 `TEST_REPORT`
6. 本轮 scope 清单
7. 已知限制与豁免说明

输入不完整时，不得伪装成“已全面体验”。应在结论中明确写出覆盖缺口，并打回 `project_manager_orchestrator`。

## 强制产物

### 1. `UX_COVERAGE_MATRIX`

覆盖账本，记录每个功能/旅程是否已按真实用户方式体验，是否完成 Trigger → Interaction → Result 评价，以及对应问题与后续动作。

### 2. `UX_REVIEW_REPORT`

高层结论文件，用于说明总体体验判断、关键断点、打回意见和放行建议。

### 3. `UX_ISSUE_LOG`

问题台账，用于持续跟踪问题状态、owner、复审情况与关闭标准。

### 4. `EXPERIENCE_ACCEPTANCE_NOTE`

只有在满足放行条件时才产出，用于正式确认可以进入 `release_preparation`。

## 执行规则

### 规则 1：先建覆盖账本，再开始评审

在真正操作前，先从以下来源枚举 release scope：

- `PRD` 的功能列表与验收标准
- `UI_SPEC` / `UX_FLOW` 的页面、组件、状态、入口
- 导航、侧边栏、详情面板、弹窗、右键菜单、悬浮工具条
- 设置、筛选、排序、搜索、批量操作、导入导出
- 空态、异常态、失败重试、撤销恢复、权限不足、网络异常
- 任何本轮宣称已支持的快捷入口或效率能力

这些条目必须先进入 `UX_COVERAGE_MATRIX`，之后才能开始逐项走查。

### 规则 2：必须建立真实用户剧本

至少构建 3 组剧本：

1. **首次用户剧本**：第一次进入，能否快速理解、找到入口、完成第一次核心任务。
2. **回访用户剧本**：带着明确目标回来，能否快速接上次状态继续做事。
3. **重度用户剧本**：高频、多步骤、批量、跨视图、连续操作时是否高效且稳定。

如果产品有明显特殊场景，还应补充：

- 异常恢复剧本
- 协作 / 分享剧本
- 设置配置剧本
- 导入导出剧本
- 跨端 / 跨视图一致性剧本

### 规则 3：每个功能都要做 Trigger → Interaction → Result 三段式核对

对每个功能条目至少记录：

- 用户通过什么入口触发
- 触发后页面/组件如何反馈
- 结果是否符合预期
- 用户是否知道发生了什么
- 如果失败，是否能恢复或得到明确引导

以下情况必须单独算体验问题：

- 入口隐蔽，用户不容易发现
- 点击后没反馈，用户不确定系统是否收到
- 结果虽成功，但表达混乱、置信度低、找不到后续动作
- 失败提示笼统，无法恢复
- 同类功能触发方式或结果表达不一致

### 规则 4：必须覆盖所有声明支持的交互方式

如果某个功能支持不止一种操作方式，不能只测最顺的一条。至少覆盖：

- 主按钮 / 次入口
- 键盘快捷键
- 拖拽 / 手势 / 上下文菜单
- 批量操作与单项操作
- 主视图与详情视图
- 正常路径与中断恢复路径

### 规则 5：评价同时覆盖结果质量与过程成本

每次评价不能只看“最后成没成功”，还要看：

- 理解成本：用户是否要停下来想
- 操作成本：步骤是否过多、顺序是否反直觉
- 反馈质量：系统是否及时、明确、可信
- 风险感受：用户会不会怀疑数据丢了、没保存、点错了
- 品质感：视觉、文案、动效、状态是否像同一个成熟产品

### 规则 6：每个问题都要落到后续处理

`UX_COVERAGE_MATRIX` 和 `UX_ISSUE_LOG` 中的每个非通过项，都必须至少写清：

- 问题级别
- 责任角色
- 下一步动作
- 是否需要复审
- 是否影响发布

缺少后续处理路径的评价，视为无效评价。

## 问题流转规则

### 角色边界

- 用户体验官只负责提出问题、说明影响、给出优化方向与复验标准，不负责具体修复，也不负责团队管理。
- 产品经理是体验问题的第一承接角色，负责把问题转成产品逻辑、范围、优先级与验收标准的正式调整。
- 项目经理 / Orchestrator 在产品经理完成问题转译后，负责组织设计、研发、QA 的返工路径、owner、时点与复审入口。
- 没有完成“体验官提问题 → 产品经理承接 → 项目经理协调 → 体验官复验”这条链路，不得把体验问题视为闭环。

### 打回给产品经理

适用于：

- 目标任务定义不清
- 功能主次关系失衡
- 验收标准不够严格
- 状态缺失导致体验口径不完整
- 文案或策略问题不是单纯界面修饰能解决的

产品经理需要把问题转成明确的需求修订、验收标准或范围调整，并把结果同步给项目经理进入返工协调。体验官不得绕过产品经理，直接把体验问题当作实现任务派给设计或研发。

### 打回给项目经理 / Orchestrator

适用于：

- 问题跨多个角色
- 需要重新安排返工优先级
- 需要暂停发布准备
- 需要组织专项复审窗口
- 有未覆盖功能或未完成复测的治理缺口

项目经理需要基于产品经理已经确认的产品逻辑与验收口径，建立返工清单、owner、时间点和复审入口，并持续维护这条返工链路直到体验官复审通过。只要 `UX_ISSUE_LOG` 里仍有阻断项，项目经理就必须保持闭环打开，不能把“已安排”视为“已关闭”。

### 标准闭环顺序

1. 用户体验官输出 `UX_REVIEW_REPORT` / `UX_ISSUE_LOG`
2. 产品经理承接问题并更新产品逻辑、范围、验收标准
3. 项目经理协调设计、研发、QA 的返工路径与复审窗口
4. 各执行角色完成实现、回归与证据补齐
5. 用户体验官基于新证据重新复验并决定是否放行

### 进入 CR / IA 的条件

若修复动作涉及以下情况，必须进入 `CR → IA`：

- 需求范围变化
- 关键交互逻辑变更
- 跨角色联动改动
- 数据结构 / 接口 / 权限 / 安全影响
- 会影响已冻结基线的行为差异

## 放行门槛

只有同时满足以下条件，才允许从 `ux_review` 进入 `release_preparation`：

1. 所有 release scope 功能都已进入 `UX_COVERAGE_MATRIX`
2. 所有核心旅程都已按真实用户剧本完整走查
3. 所有关键功能都完成 Trigger → Interaction → Result 评价
4. 没有 P0
5. 没有未对齐处理方案的 P1
6. 所有遗留风险都有明确 owner、观察点和复审策略
7. 已形成 `UX_REVIEW_REPORT`、`UX_ISSUE_LOG`
8. 满足条件后才可写 `EXPERIENCE_ACCEPTANCE_NOTE`

如果只是主链路走通、页面看起来不丑，但覆盖不完整或细节明显粗糙，仍然不能放行。

## 复审规则

复审时不允许只看“改过的那几个点”。至少要检查：

1. 上轮 P0 / P1 是否真的关闭
2. 修复是否引入新的体验倒退
3. 原来未覆盖的功能是否补测完成
4. `UX_COVERAGE_MATRIX` 是否已从 `issue` / `blocked` 更新为可接受状态
5. `UX_ISSUE_LOG` 的状态是否与真实结果一致
6. 是否已满足放行门槛

## 禁止行为

以下做法一律视为不合格评审：

- 只浏览页面，不按任务完成实际操作
- 只体验主功能，不管次级入口和边界状态
- 只记录 bug，不评价交互反馈和结果表达
- 只给审美判断，不给后续动作
- 未覆盖完整 scope 仍宣称“已全面体验”
- 因排期压力降低用户体验放行标准

## 最小治理清单

每次正式评审结束前，至少确认：

- [ ] 已建立并填写 `UX_COVERAGE_MATRIX`
- [ ] 已构建真实用户剧本
- [ ] 已覆盖 release scope 内全部功能
- [ ] 已逐项核对 Trigger → Interaction → Result
- [ ] 已对非通过项给出后续处理
- [ ] 已输出 `UX_REVIEW_REPORT`
- [ ] 已输出 `UX_ISSUE_LOG`
- [ ] 已决定是否允许写 `EXPERIENCE_ACCEPTANCE_NOTE`
