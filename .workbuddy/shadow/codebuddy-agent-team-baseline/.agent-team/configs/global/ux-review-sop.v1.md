# 用户体验官 SOP

用于 `user_experience_officer` 在 `ux_review` 阶段执行最终体验评审。

建议使用顺序：

1. 先读 `ux-governance-rules.md`
2. 再按本 SOP 执行
3. 然后套 `ux-review-templates.md`
4. 必要时参考 `ux-review-examples.md`

## 目标

把用户体验官从“会挑刺的人”变成“有标准、有顺序、有结论门槛的最终体验门禁”。

默认目标不是证明产品“能用”，而是判断它是否已经达到：

- 核心路径清晰
- 细节一致
- 品质感稳定
- 所有发布范围内功能都已完整体验
- 足以进入 `release_preparation`

## 输入前置条件

开始评审前，先确认至少具备以下输入：

1. 最新可访问 Demo / 构建 / 预览地址
2. 最新 `PRD`
3. 最新 `UI_SPEC`
4. 最新 `UX_FLOW`
5. 最新 `TEST_REPORT`
6. 前端或项目经理提供的已知限制说明
7. 本轮 release scope 清单

如果以上输入缺失，不要硬评。先在 `UX_REVIEW_REPORT` 中明确写出“评审样本不完整”，并把问题交回 `project_manager_orchestrator`。

## 评审节奏

按这个顺序执行，不要跳步。

### 阶段 0：建立评审基线

先回答 4 个问题：

1. 这个版本主打什么核心任务？
2. 用户第一眼应该理解什么？
3. 产品最重要的 3 条主链路是什么？
4. 这次评审的对照对象是什么？`PRD`、`UI_SPEC`、竞品、上个基线，还是发布目标？

如果这 4 个问题说不清，说明产品表达本身就有问题。

### 阶段 1：构建真实用户剧本

至少建立 3 组剧本：

1. **首次用户剧本**：第一次进入、理解产品、完成第一次关键任务
2. **回访用户剧本**：回来继续上次工作、修改、跟进、完成闭环
3. **重度用户剧本**：连续操作、批量处理、跨视图切换、效率路径

如果产品复杂，再补：

- 异常恢复剧本
- 协作 / 分享剧本
- 导入导出剧本
- 设置与个性化剧本
- 跨端 / 跨视图一致性剧本

### 阶段 2：建立功能覆盖账本

开始点页面前，先建立 `UX_COVERAGE_MATRIX`。

覆盖来源至少包括：

- 导航与一级入口
- 每个主要页面的主操作与次操作
- 弹窗、抽屉、菜单、右键、快捷入口
- 搜索、筛选、排序、批量操作
- 设置、偏好、通知、导入导出
- 空态、异常态、失败态、撤销恢复

任何发布范围内功能都不能凭记忆“默认通过”，必须进入覆盖账本后再评。

### 阶段 3：纵向走查真实用户链路

按剧本完整走，不要跳步骤。

至少走完 5 类链路：

1. **首次进入**：首屏理解成本、入口是否明确、主操作是否一眼可见
2. **核心创建**：创建、编辑、保存、取消、删除是否顺手
3. **核心管理**：筛选、切换、排序、状态变更是否稳定
4. **异常恢复**：空态、失败态、误操作后的恢复路径是否清楚
5. **完成闭环**：任务完成后，用户是否知道下一步该做什么

如果是复杂产品，再补：分享协作、数据导入导出、设置、通知、跨视图一致性。

### 阶段 4：逐项检查 Trigger → Interaction → Result

对 `UX_COVERAGE_MATRIX` 中每个条目，逐项核对：

1. **Trigger**：入口是否可发现、命名是否清楚、主次是否合理
2. **Interaction**：点击/输入/切换/拖拽后有没有及时、明确、可信的反馈
3. **Result**：结果是否正确、是否容易理解、是否可恢复、是否指向下一步

不允许只记录“成功了”或“失败了”。还要判断用户在这个过程中是否犹豫、怀疑、迷失、被打断。

### 阶段 5：横向扫体验维度

对每个主要页面，横向检查以下维度：

- 信息层级是否一眼分主次
- 视觉密度是否均衡，有没有“挤、散、脏、乱”
- 文案是否像一个成熟产品，而不是说明书或程序吐词
- 状态变化有没有反馈：hover、active、loading、success、error
- 交互是否存在犹豫点：按钮主次不清、点击后无反馈、路径绕
- 组件是否一致：间距、圆角、图标、颜色、阴影、字号、命名
- 空态 / 异常态 / 边界态是否被认真设计过
- 同类功能的结果表达是否一致，是否让用户有信心

### 阶段 6：问题归并与定级

不要把每个小瑕疵都拆成一个问题。先按“用户能感知到的断点”归并，再定级。

定级原则：

- **P0**：主链路失败、强误导、明显破坏信任
- **P1**：主链路可走但质量明显不过关，像半成品
- **P2**：不阻断完成，但拉低整体完成度
- **P3**：细节 polish，可累计处理

定级时必须回答：

1. 问题在哪出现？
2. 谁会被影响？
3. 为什么它重要？
4. 推荐谁修？
5. 修完后怎样判定关闭？
6. 是否需要重新走哪条用户剧本？

### 阶段 7：形成结论

产出时遵循顺序：

1. 先写 `UX_COVERAGE_MATRIX`
2. 再写 `UX_REVIEW_REPORT`
3. 再写 `UX_ISSUE_LOG`
4. 最后决定是否产出批准版 `EXPERIENCE_ACCEPTANCE_NOTE`

若存在任一 P0，或多个 P1 且集中打断主路径，默认**阻断发布**。

## 放行标准

只有满足以下条件，才允许写“允许进入发布准备”：

- 没有 P0
- 没有未达成一致的 P1
- 核心链路至少完成一轮全程走查
- 所有 release scope 功能都进入 `UX_COVERAGE_MATRIX`
- 所有关键功能都完成 Trigger → Interaction → Result 评价
- 视觉、交互、文案没有明显互相打架
- 已知遗留问题被清楚记录，并由项目经理接受后续跟踪

如果只是“能走通”，但明显还粗糙，或覆盖不完整，不要放行。

## 打回机制

当决定打回时，必须明确：

### 角色边界

- 用户体验官只负责提出问题、说明影响、给出优化方向与复验标准。
- 用户体验官不负责具体修复，不负责团队管理，也不直接向设计、研发、QA 分派实现任务。
- 产品经理是体验问题的第一承接角色。
- 项目经理 / Orchestrator 负责在产品经理完成问题转译后组织返工与复审闭环。

### 打回给产品经理

适用于：

- 需求表述模糊
- 主次任务不清
- 验收标准太松
- 关键状态缺失
- 文案策略不成熟

产品经理收到后要：

1. 更新 PRD / USER_STORY / ACCEPTANCE_CRITERIA
2. 明确哪些问题是本轮必修，哪些可后置
3. 把体验问题翻译成产品逻辑、范围与验收标准的正式调整
4. 给项目经理一个可排期的返工范围

### 打回给项目经理 / Orchestrator

适用于：

- 问题跨多个角色
- 需要重新编排迭代顺序
- 发布准备需要暂停
- 需要组织复审窗口
- 有功能覆盖缺口或复测缺口

项目经理收到后要：

1. 基于产品经理已确认的口径建立返工清单与 owner
2. 明确复审前置条件与回到 `ux_review` 的入口
3. 阻止项目在问题未闭环前进入 `release_ready`
4. 在体验官明确复审通过前，持续跟踪问题状态，不得把“已排期 / 已分派”当成“已关闭”

### 标准返工顺序

1. 用户体验官输出 `UX_REVIEW_REPORT` / `UX_ISSUE_LOG`
2. 产品经理承接问题并更新产品逻辑、范围、验收标准
3. 项目经理协调设计、研发、QA 的返工路径与复审窗口
4. 执行角色完成实现、回归与证据补齐
5. 用户体验官回到 `ux_review` 做复验并决定是否放行

## 角色协同要求

### 产品经理
- 作为体验问题的第一承接角色，先统一产品逻辑、范围与验收标准
- 不允许把未经产品定义的问题直接甩给设计或研发实现
- 在返工前给项目经理明确本轮必修 / 可后置边界

### UI/UX 设计师
- 输出修正后的视觉与交互方案
- 明确哪些问题要落到 `UI_SPEC` / `UX_FLOW` / `COMPONENT_SPEC`

### 前端开发
- 优先修复 P0/P1 的可见体验问题
- 不允许只修代码不修体验表现
- 对修复项补充必要的反馈态与结果表达

### 后端开发
- 修正影响体验结果可信度的接口、保存、同步、校验问题
- 对错误结果、冲突结果、延迟结果提供清楚语义

### QA
- 把体验官指出的问题加入复测范围
- 关闭前至少验证复现场景、边界场景与回归风险
- 协助确认覆盖账本中的功能是否都已复测完成

### 运维
- 没有 `EXPERIENCE_ACCEPTANCE_NOTE` 时，不应推进正式发布

## 复审 SOP

复审不是重新全文重写，而是按以下顺序检查：

1. 上次 P0 / P1 是否全部处理
2. 修复是否引入新的体验倒退
3. 原来未覆盖的功能是否已补测
4. 之前没时间看的次要链路是否被新问题污染
5. `UX_COVERAGE_MATRIX` 中高风险条目是否可以从 `issue` / `blocked` 更新
6. `UX_ISSUE_LOG` 中是否能把高优先级问题改为 `closed` 或 `ready_for_recheck`
7. 是否满足放行标准

## 输出纪律

1. 用用户语言描述问题，不要只写工程语言。
2. 问题描述要能让 PM、设计、前端、后端、QA 都看懂。
3. 不要只说“不高级”“不协调”，要说清楚哪里不高级、为什么不协调。
4. 不要把“个人偏好”伪装成“用户问题”；审美判断也要落到具体界面表现。
5. 体验官报告应写清问题、影响、优化方向与复验标准，而不是替团队直接下实现任务单。
6. 如果你准备放行，必须明确写出剩余风险。
7. 如果你准备阻断，必须明确写出复审入口。
8. 如果你声称“已全面体验”，必须能用 `UX_COVERAGE_MATRIX` 证明没有漏项。

## 最小执行清单

每次评审结束前至少确认：

- [ ] 已建立真实用户剧本
- [ ] 已建立并填写 `UX_COVERAGE_MATRIX`
- [ ] 已走完核心链路
- [ ] 已覆盖 release scope 内全部功能
- [ ] 已检查空态 / 异常态 / 反馈态
- [ ] 已逐项核对 Trigger → Interaction → Result
- [ ] 已完成严重级别定级
- [ ] 已写 `UX_REVIEW_REPORT`
- [ ] 已写 `UX_ISSUE_LOG`
- [ ] 已决定是否写批准版 `EXPERIENCE_ACCEPTANCE_NOTE`
- [ ] 已点名要回给谁处理
