# 用户体验官产物示例

用于快速参考输出风格。正式产物仍应以 `ux-governance-rules.md` 为覆盖边界，以 `ux-review-templates.md` 为骨架，以 `ux-review-sop.md` 为流程依据。

## 示例一：UX_COVERAGE_MATRIX（节选）

```md
# UX_COVERAGE_MATRIX — TickTick Parity Demo v2.2.0

## 1. 覆盖摘要
- release scope 功能总数：18
- 已完整体验：15
- 存在问题：3
- 未纳入本轮：0
- 覆盖结论：基本完整但有缺口

## 3. 覆盖矩阵
| Feature ID | Area | Journey | Persona | Trigger | Interaction Check | Result Check | Verdict | Issue IDs | Next Action | Owner |
|---|---|---|---|---|---|---|---|---|---|---|
| F-001 | 收件箱列表 | 首次创建 | 新用户 | 顶部“添加任务”按钮 | 输入聚焦明确，但提交成功反馈偏弱 | 任务被创建，但用户不确定是否已保存 | issue | UX-002 | 增加成功反馈并回归首次创建剧本 | 前端 |
| F-002 | 任务详情 | 回访编辑 | 回访用户 | 点击任务卡片打开详情 | 面板打开顺滑，但字段层级混乱 | 能编辑成功，但理解成本偏高 | issue | UX-001 | 重构字段分组并复测详情编辑链路 | UI/UX 设计师 |
| F-003 | 批量完成 | 重度清理 | 重度用户 | 多选后点击完成 | 反馈明确，禁用态和 loading 清楚 | 结果正确且列表状态同步稳定 | pass | — | 无 | 前端 |
| F-004 | 筛选器切换 | 回访整理 | 回访用户 | 侧栏筛选项切换 | 切换反馈存在，但当前态不够突出 | 结果正确，辨识度一般 | issue | UX-006 | 强化当前态并回归筛选链路 | 前端 |
```

## 示例二：UX_REVIEW_REPORT（节选）

```md
# UX_REVIEW_REPORT — TickTick Parity Demo v2.2.0

## 1. 审查结论
- 结论：阻断发布
- 总体完成度：7/10
- 可用性评分：8/10
- 审美一致性评分：6/10
- 覆盖完整度：8/10
- 核心判断一句话：主功能基本成型，但细节反馈和结果表达仍有明显“半成品感”，现在放行会让用户觉得产品没有收口。

## 3. 覆盖情况摘要
- 覆盖总量：15/18
- 未覆盖项：通知设置、导出确认弹窗、失败重试提示
- 高风险缺口：导出失败后的恢复路径还没有走过
- 对应覆盖矩阵：UX_COVERAGE_MATRIX--ART-UX_COVERAGE_MATRIX-0001--v1.0.0.md

## 4. 总体体验判断
### 做得好的部分
- 主工作台的信息密度已经接近目标产品
- 批量完成链路的反馈质量明显提升，重度用户效率感开始成立

### 关键断点
- 任务详情区视觉层级混乱，主次字段竞争注意力
- 快速创建完成后的结果表达不够确定，用户会怀疑自己是否保存成功
- 筛选切换后的当前态不够明确，回访用户容易多看一眼确认自己到底在哪个视图
```

## 示例三：UX_ISSUE_LOG（节选）

```md
# UX_ISSUE_LOG — TickTick Parity Demo v2.2.0

| Issue ID | Level | Status | Area | Journey | Symptom | Repro Steps | Expected | Actual | Impact | Suggested Fix | Next Action | Owner |
|---|---|---|---|---|---|---|---|---|---|---|---|---|
| UX-001 | P1 | open | 详情面板 | 回访编辑 | 字段主次混乱 | 1.打开任意任务 2.观察右侧详情 | 主字段应先被看见 | 所有输入项视觉权重接近 | 增加理解成本，像表单堆砌 | 按信息优先级重构分区 | 设计出新版分组后回归详情编辑链路 | UI/UX 设计师 |
| UX-002 | P1 | in_progress | 快速创建 | 首次创建 | 保存反馈弱 | 1.输入内容 2.回车提交 | 用户应知道任务是否创建成功 | 页面仅轻微变化，没有明确确认 | 降低信心，影响高频创建效率 | 增加动效与成功反馈文案 | 前端修复后复测首次创建与连续创建 | 前端 |
```

## 示例四：EXPERIENCE_ACCEPTANCE_NOTE（节选）

```md
# EXPERIENCE_ACCEPTANCE_NOTE — TickTick Parity Demo v2.2.1

## 1. 验收结论
- 是否通过：通过
- 结论日期：2026-03-31
- 评审角色：用户体验官
- 对应覆盖矩阵：UX_COVERAGE_MATRIX--ART-UX_COVERAGE_MATRIX-0001--v1.1.0.md
- 对应评审报告：UX_REVIEW_REPORT--ART-UX_REVIEW_REPORT-0001--v1.1.0.md

## 2. 放行依据
- 已关闭的关键问题：UX-001、UX-002、UX-006
- 剩余风险：无阻断级风险
- 可接受遗留项：
  - UX-010：统计页空态文案偏硬，但不影响核心使用
- 覆盖完整度说明：release scope 内 18 个功能均已完成真实操作，通知设置和导出失败恢复路径已补测通过

## 3. 发布建议
- 是否允许进入 `release_preparation`：是
- 发布后重点观察：
  - 高频创建链路的停留时间
  - 详情面板编辑完成率
  - 筛选切换后的回退率
```

## 怎么用这些示例

1. 用示例学习语气和颗粒度。
2. 不要机械照抄字段内容。
3. 正式写文档时先套模板，再参考示例把话说得更锋利、更可执行。
4. 如果示例和真实 scope 不一致，以真实覆盖矩阵为准，不要为了好看省略未覆盖项。
