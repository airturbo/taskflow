# TaskFlow 移动端 UX / UI 重构正式方案

> 日期：2026-04-10  
> 范围：`web/src/App.tsx`、`web/src/index.css` 为主，兼顾后续组件拆分方向  
> 目标：把当前"桌面能力压缩到手机里"的实现，重构为真正适合手机使用的任务管理体验  
> 参考方法：借鉴 **Things 3** 的低负担与流畅感、**滴答清单** 的日历与快速录入、**Notion Mobile** 的信息层级与全屏编辑思路

---

## 1. 结论先行

当前移动端问题**不是单纯样式问题**，而是四层同时失配：

1. **信息架构失配**：桌面端同时展示侧栏、视图切换、主内容、右侧详情；手机端无法承载。
2. **交互模型失配**：任务选中、详情查看、拖拽、批量操作仍带有明显桌面范式。
3. **视图策略失配**：日历、看板、矩阵仍保留桌面思路，只做了"缩窄/堆叠/强制降级"。
4. **实现结构失配**：`App.tsx` 中设备判断、视图渲染、弹层逻辑、任务逻辑耦合，导致移动端优化难以精准落地。

因此，这次移动端重构建议采用：

- **产品策略**：从"所有视图都尽量保真"转向"移动端只保留高频决策路径"。
- **交互策略**：从"点击后打开桌面式详情区"转向"卡片即入口，全屏 Sheet / 全屏详情承接深操作"。
- **结构策略**：从"宽度驱动分支"转向"设备能力 + 断点 + 场景"的统一适配层。
- **实施策略**：先统一设备与弹层模型，再改列表 / 日历 / 看板 / 矩阵，不建议继续在现有分支上堆更多手机特判。

## 2. 当前代码现状与关键问题

### 2.1 断点体系不一致

当前 `App.tsx` 中的核心判断为：

- `isPhoneViewport = viewportWidth <= 680`
- `isNavigationDrawerMode = viewportWidth <= 960`
- `isCompactSidebar = viewportWidth > 960 && viewportWidth <= 1200`
- `isUtilityDrawerMode = viewportWidth <= 1280`

但 `index.css` 仍大量使用 `@media (max-width: 720px)`，导致：

- JS 逻辑认为"不是手机"，CSS 已经进入"手机样式"；
- 交互逻辑和视觉布局不总是同一档；
- 很多问题看起来像样式 bug，实际是断点语义不一致。

### 2.2 手机端日历是"被强制 agenda"，但顶部仍保留月/周状态

当前 `CalendarView` 传参中实际存在：

- 手机端若不是 `agenda`，会被强制改成 `agenda`

这说明真实策略已经是"手机端不支持完整月/周交互"，但顶部仍让用户感知到"月 / 周 / 列表"三态，属于**状态与结果不一致**。这会直接破坏用户心智。

### 2.3 手机端"提醒 / 详情"按钮存在交互死路

当前手机顶栏仍能点击 `setUtilityDrawerOpen(true)`，但右侧 `ResponsiveDrawer` 只在：

- `isUtilityDrawerMode && !isPhoneViewport && utilityDrawerOpen`

时渲染。

也就是说，**手机上可以触发状态，但没有对应 UI 承接**。这会造成"点了没反应"的典型体验错误。

### 2.4 `TaskBottomSheet` 的关闭手势过粗

当前逻辑是：

- 记录 `touchstart`
- `touchend` 时如果向下位移超过 `80px` 就关闭

这意味着只要用户在内容区做一次较长的下滑，就可能误触关闭；它没有区分：

- 是在拖把手，还是在滚动正文；
- 当前内容是否已滚到顶部；
- 是否达到速度阈值；
- 是否处于明确的 dismiss intent。

这和 iOS / Things 3 / Notion 的 Sheet 手感差距较大。

### 2.5 移动端仍保留大量桌面式基础操作

在 `App.tsx` 中仍可看到：

- `window.prompt`
- `window.confirm`
- `onContextMenu`

这类交互在手机端几乎不可用，或体验极差。它们意味着当前清单管理、删除确认、移动操作的底层交互模型仍然偏桌面。

### 2.6 当前底部导航是"视图导航"，不是"任务流导航"

当前底部栏为：

- 日历 / 列表 / 看板 / 象限 / 统计

这对重度用户也许可理解，但不符合手机端的高频任务流。手机端更适合的是：

- 今天
- 收件箱
- 清单
- 日历
- 我的 / 更多

也就是先围绕**目标与上下文**导航，再在局部区域做视图切换。

## 3. 竞品启发与可借鉴原则

### 3.1 Things 3：低负担、强完成路径

可借鉴点：

- 底部导航承载"高频入口"，不是所有功能；
- 卡片信息非常克制，默认只展示决策所需信息；
- 滑动完成、长按进入批量/重排，比露出大量按钮更适合手机；
- 详情页承接深操作，而不是把所有字段都塞在列表里。

### 3.2 滴答清单：日历与快速录入能力强

可借鉴点：

- 移动端日历并不追求把桌面网格完整搬过来，而是强调"可浏览 + 可点选 + 可落任务"；
- 快速录入与智能时间识别，是手机端录入效率的核心；
- 月历与 agenda 联动，比单纯网格更适合小屏。

### 3.3 Notion Mobile：全屏承载与层级清晰

可借鉴点：

- 复杂对象的编辑要全屏承接，不要挤在右侧局部区；
- 通过层级、分段、折叠，而不是同时展示所有信息；
- 手机上优先保证"进入、查看、编辑"路径顺畅，再考虑多面板并存。

## 4. 推荐的移动端产品方向

### 4.1 总体设计方向

推荐采用：**静稳留白 + 卡片分层 + 手势优先**。

关键词：

- 少字
- 少按钮
- 少并列面板
- 强状态反馈
- 强主路径
- 轻量动画

不建议继续走"桌面功能全量展示，只在手机上缩小字号/堆叠布局"的路线。

### 4.2 核心原则

#### 原则 A：一屏只做一件事

手机端任何主屏最多只承载以下之一：

- 浏览任务
- 浏览日历
- 浏览一个分组 / 看板列 / 象限
- 编辑一个任务

不要在同一屏同时并列：导航 + 日历 + 详情 + 统计。

#### 原则 B：默认只展示决策信息

任务卡片默认只保留：

- 标题
- 时间 / 截止
- 优先级或清单归属中的 1 个主标签
- 完成状态

备注、协作者、附件、子任务数等改为二级信息，进入详情后再看。

#### 原则 C：拖拽必须是显式进入的模式

手机端拖拽不能"碰到就拖"，必须：

- **长按 350–450ms** 后进入拖拽态；
- 允许 `8–10px` 的移动容差；
- 进入后给放大、阴影、轻震动反馈；
- 拖拽是增强功能，不应替代"移动到…"这类显式操作。

#### 原则 D：详情要成为手机端的"操作中枢"

手机端不再保留"右侧详情区"心智，统一切为：

- 半屏可展开全屏的任务 Sheet
- 或直接全屏任务详情页

一切深操作都在这里完成。

## 5. 推荐的信息架构（v2 修订）

> **修订说明**：v1 版"今天 / 收件箱 / 清单"三个 Tab 功能边界模糊——  
> "今天"按时间筛选，"收件箱"按清单筛选，"清单"也是清单维度；用户很难一句话说清楚"今天"和"收件箱"有什么不同。  
> v2 以 **"用户打开手机最想回答的问题"** 为出发点重新拆分，并把 TaskFlow 独有的 **计划完成时间（dueAt）与硬截止（deadlineAt）双时间语义** 融入导航与卡片信息层级。

### 5.0 先理解核心数据语义

TaskFlow 每条任务最多携带三个时间字段：

| 字段 | 语义 | 用户心智 |
|------|------|---------|
| `startAt` | 计划开始 | "我打算什么时候开始做" |
| `dueAt` | 计划完成 | "我打算什么时候做完" |
| `deadlineAt` | 硬截止 / DDL | "最晚不能超过的日期，超了就出事" |

这套模型比大多数竞品（只有一个 dueDate）更丰富，移动端应把这个优势体现出来，而不是把它藏进详情页。

桌面端已经支持 `selectionTimeModes`（按 planned 或 deadline 视角切换），移动端同样需要让用户快速感知"哪些是我计划今天做的"vs"哪些是今天到 DDL 的"。

### 5.1 一级导航：基于用户核心问题拆分

手机端日程管理工具最高频的 5 个问题：

1. **"我现在该做什么？"** → 需要一个聚焦当下的入口
2. **"有没有快到 DDL 的？"** → 需要一个风险/压力维度的入口
3. **"我的日程安排长什么样？"** → 需要日历
4. **"按项目看进度"** → 需要清单/项目维度
5. **"快速记一个想法/任务"** → 需要极低摩擦的录入

据此，推荐底部 **4 Tab + 1 FAB** 布局：

```
┌──────┬──────┬──────┬──────┐
│ 焦点  │ 日历  │ 项目  │ 我的  │
└──────┴──────┴──────┴──────┘
                 ┌───┐
                 │ + │  ← 中央/右下 FAB
                 └───┘
```

#### Tab 1：焦点（Focus）

**回答的问题**：现在该做什么 + 什么快到期了。

**内容**：把原来的 "今天"、"收件箱"、"即将到来" 合并为一个**有结构的单页**，用分段而非独立 Tab 承载：

```
┌─────────────────────────────────┐
│  焦点                    [切换▾] │
├─────────────────────────────────┤
│  🔴 逾期 (2)                    │  ← 最醒目，红色区域
│  ┌─────────────────────────┐   │
│  │ 修复线上 Bug             │   │
│  │ DDL 昨天 18:00          │   │
│  └─────────────────────────┘   │
├─────────────────────────────────┤
│  📌 今天计划 (3)                │  ← dueAt = today
│  ┌─────────────────────────┐   │
│  │ 确认版本彩排清单          │   │
│  │ 计划 09:30–11:30 · DDL 12:00│ ← 同时显示 dueAt 和 deadlineAt
│  └─────────────────────────┘   │
├─────────────────────────────────┤
│  ⚡ 今天到期 (1)                │  ← deadlineAt = today 但 dueAt ≠ today
│  ┌─────────────────────────┐   │
│  │ 提交合规报告              │   │
│  │ DDL 今天 17:00           │   │
│  └─────────────────────────┘   │
├─────────────────────────────────┤
│  📥 待处理 (4)                  │  ← inbox 中无时间、未归类的任务
│  ┌─────────────────────────┐   │
│  │ 回复客户邮件              │   │
│  │ 收件箱 · 无截止           │   │
│  └─────────────────────────┘   │
├─────────────────────────────────┤
│  📅 明后天 (5)                  │  ← upcoming 2-3 天预览
│  （折叠，可展开）                │
└─────────────────────────────────┘
```

**设计要点**：

- **逾期**永远置顶，红色强调，用 `TASK_RISK_COMPARABLE_SQL` 判断（优先看 deadline，fallback 到 dueAt）。
- **今天计划** vs **今天到期** 是核心区分点：前者是"我安排今天做的"（dueAt = today），后者是"DDL 在今天但我可能没安排今天做"（deadlineAt = today && dueAt ≠ today）。这让用户一眼分清"主动计划"和"被动压力"。
- **待处理**承接原来的"收件箱"功能——不再作为独立一级 Tab，而是焦点页的一个分段。它展示 inbox 中没有排期的任务，本质是"还没决定什么时候做"。
- **明后天**作为折叠预览，给用户"后天也安排了什么"的感知，减少切日历的频率。
- 顶部 `[切换▾]` 允许在"按计划时间"和"按 DDL"两种排序视角间切换（对应已有的 `selectionTimeModes`）。

**对比 v1 方案**：

| v1 | 问题 | v2 |
|----|------|----|
| "今天" Tab | 按 dueAt 筛今天+逾期 | 合入"焦点"的"今天计划"和"逾期"分段 |
| "收件箱" Tab | 只看 inbox 清单，与"今天"视角割裂 | 合入"焦点"的"待处理"分段 |
| "清单" Tab | 与"收件箱"都是按清单维度 | 独立为"项目"Tab，专注项目维度 |

#### Tab 2：日历（Calendar）

**回答的问题**：我的时间安排长什么样。

**内容**：迷你月历 + 当日 agenda。这是独立高频场景，保留一级入口。

**与焦点的区别**：

- 焦点是**按紧迫度看任务**（逾期 → 今天 → 快到期 → 待处理）
- 日历是**按时间轴看安排**（哪天有什么、空闲在哪）

两者解决的是不同决策问题，不重复。

**日历内的二级切换**：

```
[ 月+日程 | 周 | 3日 ]
```

- **月+日程**：上方迷你月历 + 下方选中日 agenda（滴答清单模式）
- **周**：7 天横向紧凑视图，每天纵向展示任务块
- **3日**：3 天横向滑动，比周更宽松，适合密集排期用户

**时间字段的体现**：

- 日历格子上的任务条用 `startAt → dueAt` 画时间段
- 如果有 `deadlineAt`，在任务条右侧/下方用红色小标记 `DDL` 提示
- agenda 列表中同时展示"计划时间"和"截止时间"

#### Tab 3：项目（Projects）

**回答的问题**：按项目/清单维度看进度。

**内容**：展示用户创建的所有清单和文件夹，进入后可切视图。

**结构**：

```
┌─────────────────────────────────┐
│  项目                    [+新建] │
├─────────────────────────────────┤
│  📁 开始上手                     │
│     今天先跑一遍        3       │
│     发布周排期          5       │
│  📁 真实示例                     │
│     产品迭代            2       │
│     生活维护            4       │
│  ────────────                   │
│  📥 收件箱              7       │  ← 收件箱仍然在项目维度可见
└─────────────────────────────────┘
```

**进入单个清单后的二级切换**：

```
[ 列表 | 看板 | 矩阵 ]
```

- 列表 = 默认视图
- 看板 = 按状态（todo/doing/done）分列
- 矩阵 = 四象限

这样把原来底部 Tab 占满的"看板 / 象限"下沉到了项目内部的二级切换，更符合"按项目上下文切换视角"的心智。

**为什么不叫"清单"而叫"项目"**：

- "清单"给人的感觉是"一个列表"，暗示只有列表视图
- "项目"暗示"一组相关任务 + 多种看法"，更匹配实际能力
- Things 3 叫 "Projects"，滴答清单叫 "清单"但实际是项目维度
- 这个命名也给后续加"项目进度、里程碑"留了空间

#### Tab 4：我的（Me）

**回答的问题**：设置、统计、账户、同步状态。

**内容**：

- 统计概览（活跃 / 完成 / 逾期）
- 同步状态
- 主题切换
- 账户（登录 / 退出）
- 智能过滤器入口
- 标签管理入口
- 已完成 / 回收站入口

#### 中央/右下 FAB：快速创建

**回答的问题**：马上记下一个想法。

**行为**：

- 单击 → 弹出快速创建 Sheet
- 长按 → 直接语音 / 文字快速录入到收件箱（后续可扩展）

**快速创建 Sheet 的智能默认**：

- 如果当前在焦点页 → 默认 dueAt = 今天
- 如果当前在日历页且选中了某天 → 默认 dueAt = 选中日
- 如果当前在项目某个清单内 → 默认 listId = 当前清单
- 始终支持自然语言识别时间

### 5.2 二级视图：场景内切换

汇总各一级 Tab 内的二级切换：

| 一级 Tab | 二级切换 | 说明 |
|---------|---------|------|
| **焦点** | `[按计划 \| 按DDL]` | 切换 `selectionTimeModes`，改变分组排序视角 |
| **日历** | `[月+日程 \| 周 \| 3日]` | 日历布局模式 |
| **项目** (清单列表) | 无 | 项目列表页不需要视图切换 |
| **项目** (进入单个清单) | `[列表 \| 看板 \| 矩阵]` | 在项目上下文内切换视图 |
| **我的** | 无 | 设置页不需要视图切换 |

### 5.3 卡片上的时间信息展示规范

任务卡片在不同场景下展示时间的策略：

#### 焦点页

卡片展示规则：

```
标题
[计划] 09:30–11:30  [DDL] 12:00     ← 同时有 dueAt 和 deadlineAt 时
[计划] 明天 15:00                     ← 只有 dueAt 时
[DDL] 后天 18:00                      ← 只有 deadlineAt 时
[无排期]                               ← 都没有时（待处理区域）
```

- `[计划]` 用主题色标签
- `[DDL]` 用红/橙色标签
- 当 DDL 距今 ≤ 24h 时，DDL 标签变为红色闪烁 / 加粗
- 当已逾期时，整个时间行变红底

#### 日历页

- agenda 中每条任务右侧展示 `DDL` 小标记（如有）
- 月历格子上的圆点颜色区分：普通计划 = 蓝色，有 DDL = 橙色，已逾期 = 红色

#### 项目页

- 列表视图中时间行同"焦点页"规范
- 看板视图中卡片只展示最紧迫的一个时间 + DDL 标记
- 矩阵视图中不展示时间，焦点在优先级维度

### 5.4 与竞品的差异化总结

| 痛点 | 竞品通常做法 | TaskFlow 移动端方案 |
|------|------------|-------------------|
| **只有一个 due date**，分不清"我计划做"和"最晚期限" | 一个时间字段 | 计划时间 + DDL 双字段，焦点页分段展示 |
| **收件箱和今天视角割裂**，打开 App 不知道先看哪个 | 两个独立 Tab | 合为"焦点"页，按紧迫度分段 |
| **看板/矩阵塞在底部 Tab**，占了位但使用频率低 | 全局 Tab | 下沉到项目内的二级切换 |
| **日历和任务列表割裂**，看完日历还要切回列表 | 独立入口 | 日历是一级入口，焦点页有"明后天"折叠预览 |
| **快到 DDL 的任务不够醒目** | badge 或颜色 | "今天到期"独立分段 + DDL 标签全局可见 |

## 6. 分视图重构方案

### 6.1 列表视图

#### 目标

把当前"任务信息过多、操作入口过散"的列表，重构为真正适合手指浏览与快速处理的主工作流。

#### 方案

- 卡片一行主标题 + 一行次要信息；
- 默认按"逾期 / 今天 / 明天 / 本周 / 以后"分组；
- 支持右滑完成、左滑显示操作；
- 长按进入多选模式；
- 卡片点击统一进入详情 Sheet。

#### 应避免

- 卡片内堆太多元信息；
- 常驻一排按钮；
- 依赖 hover 态或右键菜单；
- 在列表直接暴露过多编辑控件。

### 6.2 日历视图

#### 目标

让手机端的日历既能"看全局日期"，又能"顺手处理当天任务"。

#### 推荐方案：混合式日历

结构分三段：

1. 顶部月份/周导航
2. 一块紧凑的 mini month / mini week 选择区
3. 下方是选中日期对应的 agenda 列表

这样能兼顾：

- 看到月度分布
- 点选日期
- 查看当天任务
- 快速新建到某天

#### 明确建议

- **不要继续在手机端强制 agenda，但顶部还保留月/周假状态**；
- 应把手机端日历定义成独立模式：
  - `calendar-mobile-hybrid`
- 如果保留月/周切换，也必须对应真实渲染，而非视觉假切换。

### 6.3 看板视图

#### 目标

在手机上保留看板的状态流转能力，但避免横向多列压缩。

#### 方案

- 顶部以 Tab 切换列；
- 当前只展示一个列内容；
- 列间切换支持左右滑；
- 卡片状态变更优先用滑动操作和"移动到"菜单；
- 长按才进入跨列拖拽。

#### 结论

手机端看板的主操作应该是：

- 滑动变状态
- 长按后拖拽
- 详情里"移动到"

而不是默认拖拽优先。

### 6.4 矩阵视图

#### 目标

保留优先级决策价值，而不是强求四宫格同时可读。

#### 方案

- 手机端默认只显示当前象限；
- 顶部用 2×2 切换器或标签切换象限；
- 当前象限任务以列表形式展示；
- 任务移动优先通过详情编辑优先级/紧急度完成。

#### 结论

矩阵在手机端更适合作为"筛选视图"，不适合作为"同时总览 4 象限"的视觉布局。

### 6.5 时间线视图

#### 目标

让时间线在手机上仍可用，但不再强依赖精密拖拽。

#### 方案

- 默认展示单日或单时间段时间轴；
- 长按任务弹出"调整时间"Sheet；
- 用时间选择器替代复杂拖拽缩放；
- 保留左右切天。

#### 结论

手机端时间线优先级低于列表与日历，可以作为 P2 阶段优化，不建议在第一阶段追求复杂可拖可拉伸的时间块编辑。

## 7. 详情与弹层模型重构

### 7.1 统一原则

手机端不再使用"右侧工具面板"概念，改为两级承载：

- **Level 1**：任务详情 Sheet（默认打开高度约 82–88vh）
- **Level 2**：全屏编辑态（当进入备注、子任务、提醒等深编辑时）

### 7.2 当前问题

当前同时存在：

- `utilityDrawerOpen`
- `taskSheetOpen`
- `selectedTaskId`

但这三者在手机与非手机之间并没有形成清晰一致的单一来源状态。

### 7.3 推荐状态模型

建议后续统一为：

- `activeTaskSurface: 'none' | 'sheet' | 'full-screen'`
- `activeTaskId: string | null`
- `activeUtilityPanel: 'none' | 'reminders' | 'details' | 'stats'`

手机端只允许：

- `sheet`
- `full-screen`

桌面端才允许：

- `drawer`
- `rail`

这样能彻底消除"手机能开 drawer 状态但没有 drawer UI"的死路问题。

### 7.4 Sheet 手势建议

关闭手势应改为：

- 仅拖动把手区域可直接触发 dismiss gesture；
- 在内容区只有当 `scrollTop === 0` 且继续下拉时才进入 dismiss；
- 同时判断位移与速度阈值；
- 若未达阈值，弹性回弹。

建议阈值：

- 位移阈值：`72px`
- 或速度阈值：`> 0.45 px/ms`

## 8. 任务卡片与触摸交互规范

### 8.1 卡片默认信息密度

默认展示：

- 标题
- 时间/截止
- 一个优先级或标签提示
- 完成勾选

折叠/隐藏：

- 长备注
- 附件清单
- 协作者列表
- 大段活动记录

### 8.2 手势优先级

推荐优先级：

1. 点击卡片：打开详情
2. 右滑：完成 / 恢复
3. 左滑：改期 / 删除 / 移动
4. 长按：进入多选或拖拽模式

### 8.3 长按拖拽规范

建议：

- 长按 `400ms`
- 移动容差 `8px`
- 触发后提供轻震动 / 视觉放大 / 阴影
- 拖拽中其余卡片让位
- 到边缘可自动滚动

同时必须保留非拖拽备选路径：

- 更多菜单 → 移动到
- 详情 → 状态 / 清单 / 象限切换

## 9. 视觉设计方向

### 9.1 推荐风格

推荐采用 **轻留白、低噪音、层级清晰** 的方向：

- 背景尽量干净；
- 卡片层级通过边框、浅阴影、微对比度建立；
- 强调色只用于当前态、主按钮、时间/优先级提示；
- 动画克制，重点服务状态切换和空间关系。

### 9.2 动效原则

只保留有意义的过渡：

- Sheet 上滑进入 / 下滑退出
- 卡片完成后淡出或轻滑出
- 视图切换轻微位移 + 淡入淡出
- 长按拖拽时卡片抬升

避免：

- 大幅旋转、夸张缩放、长时延装饰动画

### 9.3 组件气质建议

- 顶栏更轻、更薄
- 任务卡片圆角 14–16
- Sheet 顶部圆角 20–24
- FAB 尺寸清晰、可单手点按
- 所有触摸目标不小于 44px

## 10. 技术实施映射

### 10.1 第一优先级：先解决结构性错位

#### P0-1 统一断点和设备语义

先抽出一个 `useDevice` 或等价统一层，统一：

- phone
- tablet
- desktop
- touch / mouse / hybrid

并让 JS 与 CSS 共享同一套语义。

#### P0-2 收敛任务详情承载模型

优先清理：

- 手机端 `utilityDrawerOpen` 的死路径
- `selectedTaskId` / `taskSheetOpen` / `utilityDrawerOpen` 的耦合

#### P0-3 把手机端导航从"视图优先"改为"任务流优先"

即使暂时不彻底重写，也建议先完成：

- 一级入口重排
- 二级视图下沉到局部

### 10.2 第二优先级：先做高频视图

建议优先顺序：

1. 列表
2. 日历
3. 详情 Sheet
4. 看板
5. 矩阵
6. 时间线

原因：

- 列表和日历覆盖最高频使用；
- 详情是几乎所有深操作的承接点；
- 时间线在手机端收益最低，重构成本最高。

### 10.3 第三优先级：再做手势与高级交互

包括：

- 长按拖拽
- 边缘自动滚动
- 下滑关闭的物理感优化
- 多选模式
- 滑动操作
