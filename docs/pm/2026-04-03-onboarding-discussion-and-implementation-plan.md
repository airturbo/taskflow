# 2026-04-03 新手指引讨论结论与实施方案

## 结论先行

本轮新手指引不做“长篇教学页”，也不做“第一次进来强制连环弹窗”。

**最终方向：采用半引导方案** —— 先给一个克制但高级的欢迎层，再把用户带进一套经过设计的示例工作区，同时在界面里挂一份**可勾选、可跳过、可恢复**的 onboarding checklist。真正的教学发生在真实操作里，而不是说明文字里。

这更符合当前产品目标：

- 产品已经有足够多的真实能力，不需要再靠功能列表堆认知。
- 用户对完成度和审美要求高，过于用力的 tour 会立刻显得模板化。
- 这款产品的价值在“排进去、拖起来、提醒到、专注有反馈”，必须让用户亲手做一遍才能感受到。

## 项目团队讨论纪要

### 1. 项目经理视角

**目标不是“介绍功能”，而是“让新用户在 3 分钟内感到这东西真能用”。**

项目经理结论：

- 首次体验要围绕“价值激活”设计，而不是围绕信息覆盖率设计。
- onboarding 的成功标准不是看过多少提示，而是用户是否真的完成了几次关键动作。
- 需要把欢迎层、示例数据、引导状态、回看入口、完成收口都做成一条闭环，而不是零散提示。

### 2. 产品经理视角

产品侧主张：

- 引导节奏选 **欢迎层 → 示例工作区 → checklist 驱动探索 → 完成收口**。
- checklist 只保留 5 个核心动作，不能贪多。
- 每个动作都必须映射一个真实产品能力：创建、安排、拖拽、细化、反馈。
- 引导文案不要像教程，要像一个成熟产品在帮用户“快速上手自己的第一周”。

### 3. 设计 / 体验视角

设计与体验官共同结论：

- 高级感来自**克制、留白、节奏感、真实内容**，不是来自夸张插画和花哨动效。
- 首屏欢迎层应更像“welcome brief”，而不是营销页。
- 不做全局常驻高亮，不做每一步都遮罩全屏；只在当前 checklist 对应动作上给**一次聚焦提示**。
- 示例任务不能写成“点击这里试试”“拖我一下”这种低质 tutorial 文案，而要保持**真实工作语境**，由 checklist 告诉用户该去做什么。

### 4. 前端工程视角

工程侧判断：

- 不建议引入第三方导览库，容易带来模板感，也难和当前自定义交互风格统一。
- 更合适的做法是自建一套很轻的 onboarding 状态机：
  - 是否首次进入
  - 当前 step
  - step 完成状态
  - 是否跳过 / 已完成 / 稍后继续
- 引导锚点建议基于 `data-onboarding-anchor` 这类属性做，避免把定位逻辑写死在 DOM 结构里。
- onboarding 状态必须写入当前持久化体系，而不是只放临时组件 state，否则 web / desktop parity 会断。

### 5. QA 视角

QA 结论：

- 需要覆盖 web 与 desktop 两端一致性。
- 需要验证跳过、恢复、完成、重开、seed 数据已被用户修改后的退化行为。
- checklist 自动完成必须基于真实状态变化，不能只依赖用户手动打勾。

### 6. 体验官结论

体验官给出的门槛很明确：

- 新手第一次进入时，**不应该有“被上课”的感觉**。
- 看完欢迎层后，界面必须已经像一个“有人在用的真实工作台”，而不是空壳 demo。
- checklist 的存在感要低，但完成感要强。
- 做完 3 到 5 个动作之后，用户要自然理解这款产品的 4 个核心价值：
  1. 录入很快
  2. 排期很直观
  3. 调整很顺手
  4. 提醒 / 专注 /详情不是摆设

## 借鉴方向（不是照搬）

### 参考模式 A：Todoist 的“场景模板即教学材料”

可借鉴点：

- 不直接讲功能，而是给一个完整场景，让用户边看边理解产品能做什么。
- 示例内容本身就是教学载体，降低空白页焦虑。

对我们这次最有价值的启发：

> **初始化数据不只是 demo 数据，而应该是 onboarding 剧本。**

### 参考模式 B：Notion / Linear 一类产品的“渐进式引导”

可借鉴点：

- 首次进入给很轻的起步信息，不用一口气解释所有功能。
- 真正的教学放进用户正在做的事情里，通过 checklist 和 contextual cue 渐进释放。

对我们最重要的约束：

> **不要做一条从头点到尾的 tour；要让用户在真实界面里边用边懂。**

## 最终方案：半引导 onboarding 结构

## 1. 首次进入欢迎层

形式：一个单层 welcome panel，信息很少，但语气很稳。

建议结构：

- 标题：欢迎来到你的任务工作台
- 副标题：先用 3 分钟，完成 5 个关键动作，你会马上知道这套工具强在哪里
- 下方展示 3 条简洁能力摘要：
  - 快速录入与智能安排
  - 多视图拖拽与排期调整
  - 提醒、专注与详情收口
- 主按钮：开始体验
- 次按钮：先随便逛逛
- 文字入口：跳过指引

设计要求：

- 不要插画堆满屏。
- 视觉上更接近高端应用的 welcome brief，而不是落地页。
- 弹层关闭后，右上或侧边要保留“重新打开新手指引”入口。

## 2. 进入示例工作区

欢迎层结束后，默认落到一个已经排好内容的真实工作区，而不是空白页面。

默认落点建议：

- 默认选中 **列表视图**
- 默认 selection 仍可落在“今天”或一条专门设计的入门清单，但内容必须能迅速看出层次
- 默认自动选中一条“信息密度最完整”的任务，让右侧详情先处于有内容状态

这样用户一上来就能同时看到：

- 左侧导航结构
- 中间任务列表与统计摘要
- 右侧详情能力

这比先给空状态更有质感。

## 3. checklist 常驻但克制

形式建议：

- 放在主内容区右上 / summary 下方 / 侧栏一个克制卡片里
- 默认展开前 1 次，之后可折叠
- 展示 5 个 step，每个 step 都有一句短说明和一个“去完成”动作入口

### 推荐 checklist

1. **新建你的第一条任务**
   - 目标：知道顶部快速创建和自然语言录入入口
   - 完成条件：成功创建 1 条新任务
   - CTA：去顶部创建

2. **给任务安排一个时间**
   - 目标：理解日历 / 时间线不是摆设
   - 完成条件：任意任务新增 `startAt` 或 `dueAt`
   - CTA：去日历视图

3. **拖动一次任务**
   - 目标：理解看板 / 四象限 / 时间线的直接操作感
   - 完成条件：任意任务状态、象限或时间窗口发生一次拖拽变更
   - CTA：去看板试试

4. **打开详情，补全一个高级属性**
   - 目标：让用户知道详情区承载提醒、子任务、标签、评论等真实能力
   - 完成条件：新增 1 个提醒 / 子任务 / 标签中的任一项
   - CTA：打开详情

5. **完成一个专注或完成一条任务**
   - 目标：让用户感受到“执行反馈”而不是静态管理
   - 完成条件：完成 1 条任务，或完成 1 次番茄
   - CTA：开始一次专注

### checklist 设计原则

- 不让用户手动勾选；以真实行为自动完成。
- 只在当前 step 首次出现时给一个 anchored cue。
- 完成一个 step 后有轻反馈，但不要像游戏化产品一样过度兴奋。

## 4. 结束收口

当 5 个 step 完成 3 个以上时，就可视为“已激活”。

完成态建议：

- 出现一张收口卡片：
  - 标题：已经上手了，接下来把它变成你的系统
  - 主按钮：创建我的第一个真实清单
  - 次按钮：继续使用这套示例数据
- 同时保留“重新查看引导”入口，不强制消失

原因：

- 很多用户完成 onboarding 后，并不想立刻清空示例数据
- 应允许他们继续在示例工作区里改、删、替换，而不是逼着重新开始

## 初始化数据改造方案

## 核心原则

**示例数据必须服务于引导路径，而不是服务于“展示字段很多”。**

当前 seed 数据已经覆盖很多能力，但更像“功能样本集合”；下一版需要改成“带剧情的体验工作台”。

### 数据设计目标

- 打开即能看懂结构
- 每做一步都会碰到一个真实能力
- 每个视图都有东西可看，但整体不杂乱
- 任务语境真实，不像 demo 假数据

## 推荐的 seed 结构

### 文件夹 / 清单层

保留现有系统清单，但重写自定义区语义：

- 文件夹：`开始上手`
  - 清单：`今天先跑一遍`
  - 清单：`发布周排期`
- 文件夹：`真实示例`
  - 清单：`产品迭代`
  - 清单：`生活维护`

这样做的好处：

- 新手一眼就知道哪里是 onboarding 剧本，哪里是长期使用场景
- 不会把所有任务都塞在一个列表里，导致信息发散

## 推荐任务剧本

### A. 列表 / 详情主任务（默认选中）

建议放一条信息最完整的任务，例如：

- `确认周四版本彩排清单`

这条任务应自带：

- 1 到 2 个提醒
- 2 到 3 个子任务
- 1 条评论
- 1 个附件
- 1 到 2 个标签
- 已有 1 个已完成番茄记录

用途：

- 用户一进来就能看到详情不是空的
- 右侧详情能完整展示这款产品“不是只能记标题”

### B. 看板演示任务

准备 3 到 4 条同一项目任务，分布在 todo / doing / done。

例如：

- 整理发布说明
- 补齐活动素材
- 确认首发节奏
- 复核公告文案

用途：

- 用户切到看板时立刻理解“这不是另一个列表皮肤”，而是真能拖状态

### C. 日历 / 时间线演示任务

至少准备 2 条有明确时间窗的任务：

- `周二 15:00 设计评审`
- `周三 10:00 - 12:00 发布彩排`

用途：

- 月 / 周 / agenda / timeline 都有内容可看
- 用户只要改一次时间，就能理解排期回写

### D. 四象限演示任务

显式准备带“紧急 / 重要”特殊标签的任务组合：

- 紧急且重要：今天必须给结论
- 重要不紧急：下周方案预研
- 紧急不重要：同步外部素材
- 不紧急不重要：归档旧视觉稿

用途：

- 让用户一眼理解四象限由标签驱动，而不是黑箱优先级

### E. 提醒 / 通知演示任务

至少保留 1 条会很快触发提醒的任务，例如未来 10 到 20 分钟内。

用途：

- 新用户在 onboarding 期间就有机会感受到真实提醒，而不是只看到配置项

### F. 已完成 / 回收站演示任务

继续保留 1 条 completed、1 条 deleted。

用途：

- 用户切换到已完成 / 回收站时不至于空
- 同时说明系统具备完整生命周期，而不是只有当前列表

## 实施建议

## 1. 状态模型

建议在 `PersistedState` 中新增 `onboarding` 字段，统一走现有持久化链路。

建议结构：

```ts
interface OnboardingState {
  version: 'v1'
  status: 'not_started' | 'in_progress' | 'completed' | 'dismissed'
  currentStepId: string | null
  completedStepIds: string[]
  lastSeenAt: string | null
  seedScenarioVersion: 'v1'
}
```

原因：

- web / desktop 共享
- 可以安全做回看、恢复、版本升级
- 后面若需要第二版 onboarding，可按 version 演进

## 2. 触发条件

首次展示不要只靠“本地有没有值”，要判断是不是**真正的新工作区**。

建议判定逻辑：

- onboarding 状态未完成、未跳过
- 当前数据仍符合 seed 场景签名
- 用户尚未形成自己的大规模数据修改

也就是说：

- 新用户进来会触发
- 老用户即使升级版本，也不会突然被 onboarding 打断

## 3. 引导组件实现策略

不引第三方库，自己做 3 个轻量组件即可：

- `WelcomePanel`
- `OnboardingChecklist`
- `ContextCuePopover`

并给关键交互点补 anchor：

- `data-onboarding-anchor="quick-add"`
- `data-onboarding-anchor="calendar-create"`
- `data-onboarding-anchor="kanban-column"`
- `data-onboarding-anchor="detail-reminder"`
- `data-onboarding-anchor="focus-start"`

这样后续即使 UI 继续微调，也不会把 onboarding 逻辑打散。

## 4. 自动完成逻辑

每个 step 都要绑定真实行为信号：

- 创建任务成功
- 任意任务时间发生变化
- 任意拖拽写回成功
- 详情属性新增成功
- 完成任务 / 完成番茄

不要提供纯手动勾选开关。

## 5. seed 数据版本化

建议给 seed 场景加一个版本号或 marker。

目的：

- 后续可以升级示例数据
- 只对未被用户深改的工作区进行平滑升级
- 避免覆盖用户已经开始真实使用的数据

## 推荐实施顺序

### Phase 1：打基础

- 扩展 `PersistedState` 的 onboarding 状态
- 打通 web / desktop 持久化
- 做首次进入判定

### Phase 2：做壳层

- 落 welcome panel
- 落 checklist 卡片
- 落 contextual cue 能力

### Phase 3：重写 seed 数据

- 把现有 seed 从“功能样本”改成“引导剧本”
- 调整默认选中任务与默认落点
- 确保列表 / 日历 / 看板 / 时间线 / 四象限 / 已完成 / 回收站都有合理首屏内容

### Phase 4：接自动完成

- 给 5 个 checklist step 接真实行为判断
- 接完成收口卡片
- 加“重新查看引导”入口

### Phase 5：走 UX / QA 门禁

- 新用户首轮体验走查
- 跳过 / 恢复 / 已修改 seed 的回归
- 桌面端提醒 / 声音 / 番茄反馈复测

## 验收标准

完成后至少要满足下面这些标准：

### 体验标准

- 首次进入 10 秒内能看懂产品大致结构
- 不会被连续弹窗教育
- 3 分钟内能完成至少 3 个关键动作
- 每个动作完成后都能感知到“系统真的回写了”
- 整体视觉高级、克制，不像模板引导库

### 功能标准

- onboarding 可跳过、可恢复、可完成
- checklist 自动完成准确
- web / desktop 状态一致
- seed 数据与 onboarding 步骤一一对应
- 用户修改过示例数据后，不会被强制重置

### 体验官门禁标准

- 新手引导不是单独存在的一层壳，而是和真实产品能力严丝合缝
- 不允许出现“为了引导而引导”的假动作
- 不允许出现写得很像 demo 的示例任务标题

## 我建议的落地口径

如果下一步直接开始实现，我建议按下面这个口径推进：

**第一版先把 onboarding 做到“克制但完整”，而不是一上来追求复杂动画。**

也就是先确保：

1. 欢迎层质感对
2. checklist 逻辑通
3. seed 剧本真的能教会用户
4. 自动完成可信
5. 完成后能自然转入真实使用

这 5 件事做对了，这套 onboarding 就已经会很高级；反过来，如果先追花哨动效，很容易把东西做廉价。
